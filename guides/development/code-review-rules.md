## 📅 文档信息
- **创建时间**: 2026-03-29
- **最后更新**: 2026-03-29

---

## 🎯 规则概述

### 核心原则
**所有代码变更在合并前必须通过专业的代码审查和架构审查。**

### 为什么需要这个规则？
1. **保证代码质量** - 避免技术债务积累
2. **确保架构合理** - 防止架构恶化
3. **提升可维护性** - 保持代码清晰易懂
4. **减少Bug** - 及早发现潜在问题
5. **知识共享** - 通过审查传播最佳实践

---

## 📋 审查触发条件

### 必须审查的情况（强制）
- ✅ **所有涉及架构变更的代码**
- ✅ **新增或修改API层代码**
- ✅ **修改核心业务逻辑**
- ✅ **引入新的依赖或库**
- ✅ **性能敏感的代码**
- ✅ **安全相关的代码**

### 建议审查的情况
- 🔵 重构现有代码
- 🔵 修复复杂Bug
- 🔵 添加新功能模块
- 🔵 修改公共接口

### 可以跳过审查的情况
- ⚪ 修复明显的拼写错误
- ⚪ 简单的样式调整
- ⚪ 更新文档或注释
- ⚪ 修改测试文件

## 🔍 审查维度

### 1. 代码复用审查（Reuse Review）

**检查要点**：
- [ ] 是否重复了现有的工具函数？
- [ ] 是否有相似的功能可以合并？
- [ ] 是否使用了现有的类型定义？
- [ ] 是否应该使用共享的配置或常量？

**常见问题**：
- ❌ 重复创建Supabase客户端（应该使用共享实例）
- ❌ 重复的CRUD模板代码（应该使用Base类）
- ❌ 重复的错误处理逻辑（应该统一）

### 2. 代码质量审查（Quality Review）

**检查要点**：
- [ ] 是否违反SOLID原则？
- [ ] 是否违反DRY原则？
- [ ] 是否有重复的状态管理？
- [ ] 是否会导致测试困难？
- [ ] 是否泄露了抽象？
- [ ] 是否有"stringly-typed"代码？

**关键原则**：
- **S - 单一职责原则（SRP）**：每个类只负责一件事
- **O - 开闭原则（OCP）**：对扩展开放，对修改关闭
- **L - 里氏替换原则（LSP）**：子类可以替换父类
- **I - 接口隔离原则（ISP）**：不应该依赖不需要的接口
- **D - 依赖倒置原则（DIP）**：依赖抽象而非具体实现

### 3. 效率审查（Efficiency Review）

**检查要点**：
- [ ] 是否有不必要的计算或重复工作？
- [ ] 是否错过了并发优化的机会？
- [ ] 是否在热路径上添加了阻塞操作？
- [ ] 是否有内存泄漏风险？
- [ ] 是否有N+1查询问题？
- [ ] 是否做了不必要的存在性检查？

**性能阈值**：
- 启动路径操作 < 10ms
- 内存增长 < 1MB
- 数据库查询优化（使用JOIN避免N+1）

### 4. 架构审查（Architecture Review）

**检查要点**：
- [ ] 是否符合项目的架构模式？
- [ ] 是否破坏了现有的抽象层次？
- [ ] 是否引入了不必要的耦合？
- [ ] 是否考虑了未来的扩展性？
- [ ] 是否遵循了现有的命名约定？

**架构原则**：
- 分层清晰（Presentation → Business → Data Access）
- 依赖方向正确（高层依赖低层，不反向）
- 模块化良好（高内聚，低耦合）

### 5. 安全审查（Security Review）

**检查要点**：
- [ ] 是否暴露了敏感信息？
- [ ] 是否有注入攻击风险（SQL、XSS等）？
- [ ] 是否正确处理了用户输入？
- [ ] 是否有认证/授权问题？
- [ ] 是否依赖了不安全的第三方库？

## 📝 审查流程

### 标准流程

1. **代码变更**
   ```bash
   git checkout -b feature/your-feature
   # 进行代码修改
   git add .
   git commit -m "feat: your changes"
   ```

2. **自动审查**
   ```bash
   # 运行所有检查
   npm run lint
   npm run type-check
   npm run test
   npm run build
   ```

3. **专业审查**
   ```bash
   # 使用simplify skill进行综合审查
   /simplify
   ```

4. **架构审查**
   - 如果涉及架构变更，使用专门的architecture skills
   - 确认架构变更的合理性

5. **修复问题**
   - 根据审查意见修改代码
   - 重新运行检查

6. **确认通过**
   - 所有自动化检查通过
   - 专业审查无重大问题
   - 架构审查确认合理

7. **合并代码**
   ```bash
   git checkout main
   git merge feature/your-feature
   ```

## 🚨 红线规则（必须遵守）

### 禁止的做法
1. ❌ 跳过架构变更的审查
2. ❌ 直接在生产环境修改代码
3. ❌ 提交有linting错误的代码
4. ❌ 提交有类型错误的代码
5. ❌ 引入未审查的第三方依赖
6. ❌ 硬编码密钥或敏感信息
7. ❌ 违反现有的架构模式

### 必须的做法
1. ✅ 所有架构变更需要专业审查
2. ✅ 所有API层修改需要审查
3. ✅ 使用依赖注入而非硬编码依赖
4. ✅ 共享资源使用单例模式
5. ✅ 错误处理不泄露实现细节
6. ✅ 使用类型安全的代码
7. ✅ 编写适当的测试

## 📚 案例研究

### 案例1：API架构审查（2026-03-29）

**问题**：
- 4个API类各自创建Supabase客户端
- 违反DRY和DIP原则
- 代码重复率60-70%

**审查结果**：
- 代码复用：⭐⭐☆☆☆ (2/5)
- 代码质量：⭐⭐⭐☆☆ (3.5/10)
- 效率：⭐⭐⭐⭐⭐ (0.6/10影响)

**改进方案**：
- 使用单例模式创建共享客户端
- 创建BaseAPI抽象类
- 通过继承减少重复代码

**收益**：
- 代码重复率：60-70% → <10%
- 可测试性：2/10 → 8/10
- 维护成本：显著降低

## 💡 最佳实践

### 代码审查Tips
1. **保持开放心态** - 审查是为了改进代码，不是批评个人
2. **提供具体建议** - 不仅要指出问题，还要提供解决方案
3. **关注关键问题** - 优先解决架构和质量问题
4. **尊重原作者** - 讨论时要尊重原代码的设计意图
5. **持续学习** - 通过审查学习新的模式和技巧
