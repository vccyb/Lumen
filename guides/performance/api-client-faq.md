## ❓ 常见问题

### Q1: 创建4个Supabase客户端是否会创建4个HTTP连接？

**A: 不会。**

现代浏览器使用HTTP/2或HTTP/1.1连接池，同一域名下的请求会自动复用TCP连接。虽然你有4个客户端实例，但它们都连接到同一个Supabase域名，浏览器会复用连接。

**验证方法**:
```javascript
// 在浏览器控制台
// 打开DevTools -> Network标签
// 观察所有Supabase请求，它们会共享同一个TCP连接
```

---

### Q2: 这会导致数据库连接池问题吗？

**A: 不会。**

数据库连接池在服务器端（Supabase的基础设施），由PgBouncer管理。浏览器端的客户端实例数量不影响服务器端的连接池。

**架构说明**:
```
浏览器客户端 → Supabase API → PgBouncer连接池 → PostgreSQL
              ↑                ↑
           4个实例          这里管理连接池
           (无影响)         (不受影响)
```

---

### Q3: 4个客户端会增加多少内存使用？

**A: 约4-8 KB。**

每个客户端实例占用约1-2 KB内存。在典型的React应用（50-150 MB）中，这个占比是**0.005% - 0.016%**，完全可以忽略。

**详细分解**:
- URL和认证信息: ~200 bytes
- 内部配置对象: ~1 KB
- Auth订阅管理器: ~500 bytes
- Realtime客户端（未启用）: ~1-2 KB

---

### Q4: 每个客户端维护独立的缓存会影响效率吗？

**A: 轻微影响，但不严重。**

**受影响的缓存类型**:
1. **认证状态**: 每个客户端首次调用`auth.getSession()`时会发起HTTP请求，之后才缓存
2. **Realtime订阅**: 如果启用，每个客户端会创建独立的WebSocket连接（这是更大的问题）

**实际影响**:
- 应用首次加载时可能有4次重复的认证检查
- 每次约100-200ms，总共约400-800ms的额外延迟
- 但这只在首次加载时发生，之后使用缓存

---

### Q5: 是否存在N+1查询问题？

**A: 当前不存在。**

代码审查显示所有API类都使用了正确的查询模式：
- MilestoneAPI: 使用JOIN查询 `.select('*, milestone_tags(tag)')`
- 其他API: 简单的`.select('*')`查询，无关联查询

**如果添加关联查询，需要注意**:
```typescript
// ❌ 错误示例（可能导致N+1）
const milestones = await getAll();
for (const m of milestones) {
  const tags = await getTags(m.id); // N次额外查询
}

// ✅ 正确示例（使用JOIN）
const milestones = await supabase
  .from('milestones')
  .select('*, milestone_tags(tag)'); // 1次查询
```

### Q6: 启动时是否有明显的性能开销？

**A: 可忽略。**

每个`createClient()`调用耗时约1ms，4个实例约4ms。在典型的Next.js应用启动时间（500-2000ms）中，占比**0.2% - 1%**。

**性能分解**:
```
应用启动总时间: 500-2000ms
├── JavaScript解析: 300-1000ms
├── React渲染: 100-500ms
├── API客户端初始化: 4ms (0.2%-1%)
└── 其他: 96-1496ms
```

---

### Q7: 为什么建议使用单例模式？

**A: 主要原因不是性能，而是可维护性。**

**单例模式的优势**:
1. ✅ **统一缓存**: 避免重复的认证检查
2. ✅ **代码简化**: 消除重复的构造函数代码
3. ✅ **易于维护**: 客户端配置集中管理
4. ✅ **Realtime就绪**: 未来启用Realtime时只有一个WebSocket连接
5. ✅ **最佳实践**: 符合Supabase和JavaScript社区的推荐模式

**性能提升是次要的**（只节省约8KB和3ms），但代码质量提升是主要的。

---

### Q8: 什么时候必须使用单例模式？

**A: 启用Realtime功能时。**

如果你使用Supabase Realtime（实时订阅），每个客户端会创建一个独立的WebSocket连接：
- **当前设计**: 4个客户端 = 4个WebSocket连接 ❌
- **单例模式**: 1个客户端 = 1个WebSocket连接 ✅

**WebSocket连接的问题**:
- 每个连接占用服务器资源
- 浏览器对同一域名的连接数有限制（通常6个）
- 多个连接会增加电池消耗（移动设备）

**A: 在当前阶段，影响可忽略。**

**性能影响评分** (0-10分，10为最严重):
- HTTP连接浪费: 0/10 (无影响)
- 内存开销: 1/10 (影响极小)
- 启动开销: 1/10 (可忽略)
- 数据库连接池: 0/10 (无影响)
- 缓存隔离: 2/10 (轻微影响)
- 网络开销: 0/10 (无影响)
- N+1查询: 0/10 (不存在)
- **总体评分**: 0.6/10 (几乎无影响)

**结论**: 这是一个代码质量问题，不是性能问题。
